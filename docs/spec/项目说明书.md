# 项目说明书：TeamFilePermissionCenter（团队比赛项目文件权限中心）

> 目标：在内网环境中，通过 Web 管理团队成员与比赛项目文件，并提供**文件级权限控制**、**审计追踪**、**版本与备份**、**回收站防误删**、以及“个人区（同项目内对他人不可见）”能力。

## 1. 背景与目标

### 1.1 解决的问题

- 团队多人协作时，比赛资料（程序系统、PPT、计划书、成绩单、专利材料等）分散且权限难控。
- 需要在项目内实现“**谁能看、谁能下、谁能改**”，并能追溯行为。
- 需要避免误删，并支持项目期间的定期备份留存。
- 需要在不增加服务器负载的前提下，提供 PDF 等文档的良好预览体验（OCR 暂不做）。

### 1.2 范围（MVP + 可扩展）

**MVP（第一阶段）**
- 账号登录与角色
- 团队/项目/成员管理
- 项目文件树浏览、文件上传/下载/预览、重命名/移动/删除
- 文件级权限（含目录继承 + 文件覆盖）
- 审计与追踪（读/写/权限变更）
- 回收站（软删除、恢复、到期清理）
- 版本记录（至少元数据版本；可扩展到内容版本库）
- 全文检索（先支持可抽取文本的 PDF/文本；不做 OCR）

**后续可扩展（按需）**
- 审批流：加入项目、申请访问某文件/目录、申请下载权限
- 文档转换缓存策略增强（Office → PDF 预览）
- 更细的防泄露策略（水印、分享链接、下载次数限制等）

## 3. 核心概念与权限模型

### 3.1 角色（Roles）

- **系统管理员（Admin）**：全局管理；创建项目；管理成员；管理所有项目权限与文件。
- **项目管理员（ProjectAdmin）**：仅管理所辖项目；可管理项目成员与权限；默认可见个人区（按你的选择：模式 2）并可管理。
- **项目成员（Member）**：可访问自己被授权的目录/文件；默认可见自己的个人区内容。
- **只读访客（可选）**：仅可浏览/预览特定公开目录（如后续需要）。

> 备注：你要求“项目管理员/管理员默认可见且可管理个人区（模式 2）”，并在界面提供“显示/不显示个人区”的切换。

### 3.2 权限集合（Actions）

建议的权限粒度（可按项目裁剪）：
- `LIST`：查看目录列表（可见性）
- `PREVIEW`：在线预览（PDF/图片/文本）
- `DOWNLOAD`：下载文件
- `UPLOAD`：上传/新建文件
- `UPDATE`：覆盖更新文件内容
- `MOVE`：移动/改名
- `DELETE`：删除（进入回收站）
- `RESTORE`：从回收站恢复
- `ACL_ADMIN`：管理文件/目录权限（ACL）

### 3.3 权限继承与覆盖

见：[diagrams/权限继承与覆盖图.md](../diagrams/权限继承与覆盖图.md)

规则建议：
- 默认：目录 ACL 继承到子目录/文件。
- 文件可设置“显式 ACL”，用于覆盖继承结果。
- 冲突优先级：**显式拒绝 > 显式允许 > 继承允许**（更安全）。

### 3.4 个人区（同项目内对他人不可见）

你新增要求：个人区文件“直接放在项目中”，但对其他成员不可视、本人可见。

推荐实现：
- 对 `FileAsset` 增加标记：`is_personal=true` 与 `owner_user_id`。
- 默认可见规则：
  - owner：可见（按其 ACL/角色进一步限制写操作亦可）
  - Admin/ProjectAdmin：可见且可管理（你已选择“模式 2”）
  - 其他成员：不可见（`LIST` 阶段就过滤掉）
- UI：在项目文件页提供开关：
  - “显示个人区” / “隐藏个人区”
  - 开关只影响列表展示，不改变权限本身。

个人区策略图见：[diagrams/个人区访问策略图.md](../diagrams/个人区访问策略图.md)

## 5. 数据模型（ER 图 + 关键字段）

ER 图见：[diagrams/ER图.md](../diagrams/ER图.md)

关键实体建议（摘要）：
- `User`：账号、状态、认证信息
- `Project`：项目、根目录、状态（进行中/归档/结束），以及项目编码/描述/起止日期/比赛阶段/进度
- `ProjectMembership`：用户与项目关系 + 角色
- `FileAsset`：文件/目录元信息（相对路径、类型、hash、owner、is_personal）
- `FileVersion`：版本记录（变更事件、hash、存储位置、操作者、时间）
- `FileACL`：对文件/目录授予的权限条目（允许/拒绝、继承标记）
- `RecycleItem`：回收站条目（原路径、删除人、删除时间、到期时间）
- `AuditLog`：审计日志
- `IndexJob`：索引任务（文件变更触发）
- `BackupJob`：备份任务与结果
- `ApprovalRequest`（预留）：审批流

### 5.1 项目“比赛进度/奖项”字段（用于首页/项目管理右侧详情）

- `Project.code`：项目编码（例如 PRJ2025...），用于展示与检索
- `Project.description`：项目描述（右侧详情“描述”区域）
- `Project.start_date` / `Project.end_date`：项目起止日期（允许 end_date 为空）
- `Project.competition_stage`：当前比赛阶段（建议枚举：`SCHOOL|CITY|PROVINCE|NATIONAL`）
- `Project.progress_percent`：当前进度（0-100，用于进度条）
- `ProjectAward`：阶段奖项记录（可多条），字段建议：stage/title/level/awarded_at/description

## 6. 文件操作、回收站与版本

### 6.1 文件操作边界

- 所有路径以项目根目录为沙箱，统一做“规范化 + 防穿越”。
- 目录/文件展示阶段即按 `LIST` 权限过滤，避免“看得到但点不开”。

### 6.2 回收站（减少误删）

需求要点：回收站入口要多，减少页面跳转；入口 API 复用。

建议产品行为：
- 删除默认进入回收站（软删除），不立即物理删除。
- 回收站支持：搜索、按删除人/时间筛选、批量恢复、批量清理。
- 回收站“入口多处出现”（不新增复杂页面，仅多个入口指向同一回收站视图/API）：
  - 项目文件页工具栏固定入口
  - 左侧导航入口
  - 文件列表空状态按钮入口
  - 文件/目录右键菜单入口

### 6.3 版本管理

- 每次写入行为（UPLOAD/UPDATE/MOVE/DELETE/RESTORE）生成 `FileVersion` 事件。
- 内容版本策略（可选）：
  - 小文件：保留历史内容副本
  - 大文件：只保留元数据版本 + 外部备份兜底

## 7. 审计与追踪

审计必须覆盖：
- 登录/登出、权限变更、成员变更
- 浏览目录、预览文件、下载文件
- 上传/覆盖更新、移动/改名、删除/恢复
- 备份/恢复、索引任务触发与结果

审计建议字段：
- `actor_user_id`、`project_id`、`path`、`action`、`result`、`reason`、`ip`、`user_agent`、`created_at`

## 8. 全文检索与低负载预览（无 OCR）

- OCR 暂不支持：扫描件需要识别时，由用户下载后自行处理。
- PDF 预览：推荐前端使用 PDF.js；后端支持 Range 分片读取。
- 索引策略：后台抽取文本（PDF/纯文本/可选 Office 转 PDF 后抽取）写入索引。
- 权限过滤：搜索结果必须二次按 `LIST/PREVIEW` 权限过滤，避免“搜到不可见文件”。

## 10. 非功能需求（NFR）

- 性能：目录列表分页；大文件下载/预览使用 Range；索引异步。
- 可用性：备份可恢复；回收站防误删。
- 安全：路径沙箱；鉴权一致；最小权限；审计留痕。
- 部署：Linux 上通过 Nginx 反代；后端可用 Gunicorn/Uvicorn；数据库优先 PostgreSQL（也可先 SQLite）。
