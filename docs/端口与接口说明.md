# 端口与接口说明（IP:端口访问）

本文档说明本项目部署后各端口用途，以及后端 `8000` 端口提供的管理/业务路径。

> 注意：文档中所有 `<SERVER_IP>` 需要替换成你的服务器公网 IP。

---

## 1. 端口用途总览

| 端口 | 服务 | 用途 | 访问方式 | 备注 |
|---:|---|---|---|---|
| 22 | SSH | 远程登录与运维 | `ssh ubuntu@<SERVER_IP>` | 云安全组一般已开 |
| 5173 | Web 前端 | 用户操作界面（静态站点） | `http://<SERVER_IP>:5173` | 推荐使用 `serve -s` 支持 SPA 路由 |
| 8000 | 后端 API（Django/Gunicorn） | 业务 API、后台管理 | `http://<SERVER_IP>:8000` | 根路径 `/` 可能为 404（正常） |
| 9000 | MinIO API | 文件上传/下载/预览（预签名 URL） | `http://<SERVER_IP>:9000` | 浏览器直接打开通常返回 400（正常） |
| 9001 | MinIO Console | MinIO 管理控制台 | `http://<SERVER_IP>:9001` | 生产可不开放 |

---

## 2. 快速验证每个端口是否正常

在服务器本机执行：

```bash
# 前端
curl -I http://127.0.0.1:5173/

# 后端 API（401/403 均正常，表示服务在）
curl -I http://127.0.0.1:8000/api/v1/

# MinIO API（400 也算正常）
curl -I http://127.0.0.1:9000/

# MinIO Console（200/30x 正常）
curl -I http://127.0.0.1:9001/
```

查看端口是否监听：

```bash
sudo ss -lntp | egrep '(:5173|:8000|:9000|:9001)\b' || true
```

## 3. `8000` 端口（后端）路径说明

后端由 Django 提供，项目路由前缀如下：

- Django Admin：`/admin/`
- 业务 API 前缀：`/api/v1/`

### 3.1 根路径 `/` 为什么是 404？

访问 `http://<SERVER_IP>:8000/` 显示 Not Found 通常是正常的：
- 该项目没有为根路径 `/` 提供页面或说明。
- 正确的 API 前缀是 `http://<SERVER_IP>:8000/api/v1/`。

### 3.2 Django Admin（后台管理）

- `GET http://<SERVER_IP>:8000/admin/`
  - Django 自带后台（可用于查看/管理数据库对象）
  - 需要使用 `createsuperuser` 创建的超级管理员登录

### 3.3 认证与 Token

API 前缀：`http://<SERVER_IP>:8000/api/v1`

- `POST /auth/login/`
  - 获取 JWT：返回 `access` 与 `refresh`
  - 请求体：`{"username": "...", "password": "..."}`

- `POST /auth/refresh/`
  - 刷新 JWT

- `GET /auth/me/`
  - 获取当前用户信息（需要 `Authorization: Bearer <access>`）

### 3.4 项目与成员

- `GET /projects/`：项目列表
- `POST /projects/`：创建项目（需要系统管理员/项目管理员权限）
- `GET /projects/{id}/`：项目详情
- `PATCH /projects/{id}/`：更新项目
- `GET /projects/{id}/members/`：项目成员列表
- `POST /projects/{id}/members/`：添加成员

### 3.5 文件中心（核心）

`files` 相关接口均以 `/files/` 为前缀（这些接口会进行项目成员校验与 ACL 权限校验）。

- `GET /files/tree/?project=<projectId>&include_personal=0|1`
  - 获取目录树

- `GET /files/?project=<projectId>&folder=root|<folderId>&include_personal=0|1`
  - 获取当前目录列表

- `POST /files/folders/`
  - 新建文件夹

- `POST /files/upload/`（multipart）
  - 上传文件

- `GET /files/{id}/download/`
  - 获取下载预签名 URL

- `GET /files/{id}/preview/`
  - 获取预览预签名 URL

- `POST /files/{id}/update-content/`（multipart）
  - 覆盖更新文件内容（会产生版本记录）

- `GET /files/{id}/versions/`
  - 查看文件历史版本

- `POST /files/{id}/rename/`
  - 重命名

- `POST /files/{id}/move/`
  - 移动

- `POST /files/{id}/delete/`
  - 软删除（进入回收站）

- `GET /files/recycle/?project=<projectId>`
  - 文件回收站列表

- `POST /files/{id}/restore/`
  - 从回收站恢复

- `DELETE /files/{id}/purge/`
  - 从回收站彻底删除

- `GET /files/search/?project=<projectId>&q=<keyword>&include_personal=0|1`
  - 文件名/内容搜索（PDF/文本抽取）

### 3.6 权限（ACL）

- `GET /files/acl/?project=<projectId>&target=root|<fileEntryId>`
  - 查看某目标的 ACL 条目（需要项目管理员或系统管理员）

- `POST /files/acl/grant/`
  - 创建 ACL 条目（允许/拒绝、权限集合、是否继承）

- `DELETE /files/acl/{acl_id}/`
  - 删除 ACL 条目

### 3.7 审计

- `GET /audit/?project=<projectId>&action=<ACTION>`
  - 获取审计日志

### 3.8 系统备份（管理员）

- `GET /backups/`：列出备份文件
- `POST /backups/create/`：创建备份
- `POST /backups/restore/`：恢复备份

## 4. 常见问题

### 4.1 前端能打开但登录失败

常见原因：前端构建时没有设置 `VITE_API_BASE`，导致浏览器请求错误地址（例如指向本机 `127.0.0.1`）。

解决：重新构建并写死 API 地址：

```bash
cd /home/ubuntu/web
VITE_API_BASE=http://<SERVER_IP>:8000/api/v1 npm run build
sudo systemctl restart tfpc-web
```

### 4.2 访问 `http://<SERVER_IP>:5173/login` 404

若使用 `python3 -m http.server` 静态服务，SPA 路由会 404。
推荐使用 `serve -s dist` 方式部署静态站点。
