# 项目代码审查与后续规划 (Code Review & Roadmap)

## 1. 当前完成情况 (MVP Status)

经过审查，项目的第一阶段 (MVP) 功能已全部实现。

### 1.1 核心功能
- **文件管理**: 上传、下载、预览 (PDF/文本)、移动、重命名、删除。
- **权限控制 (ACL)**: 基于 RBAC (角色) 和 ACL (访问控制列表) 的细粒度权限，支持继承与覆盖。
- **个人区**: 支持“个人区”文件标记，同项目内对他人不可见。
- **回收站**:
  - **文件回收站**: 支持软删除、还原、彻底删除。支持显示原路径。
  - **项目回收站**: 支持项目删除后的恢复。
- **版本管理**: 文件覆盖更新时自动生成版本记录，支持查看历史版本列表。
- **全文检索**: 支持文件名搜索，以及 PDF/文本文件的内容全文检索 (基于 `pypdf` 提取文本)。
- **系统备份**: 支持手动触发系统级数据备份与恢复。
- **审计日志**: 覆盖了文件操作、权限变更、项目管理等关键行为。

### 1.2 代码结构
- **后端 (Django)**:
  - `models.py`: 数据模型设计符合 ER 图，包含 `FileEntry`, `FileVersion`, `FileACL`, `RecycleBinItem` 等核心表。
  - `views.py`: 业务逻辑完整，包含权限校验 (`_require_acl`)、文本抽取 (`_extract_text`)、递归操作 (删除/恢复子树) 等。
  - `serializers.py`: 序列化逻辑清晰，`get_original_path` 实现了回收站路径还原。
- **前端 (Vue 3)**:
  - `FilesPage.vue`: 实现了完整的文件浏览、搜索、回收站视图、历史版本对话框。
  - `SettingsPage.vue`: 实现了备份管理界面。

## 2. 建议添加的功能 (Future Extensions)

根据项目说明书与代码现状，以下功能建议作为下一阶段的开发目标：

### 2.1 审批流 (Approval Workflows)
- **现状**: 目前权限由管理员直接分配。
- **建议**: 添加 `ApprovalRequest` 模型。
  - **加入项目申请**: 用户申请加入项目，管理员审批。
  - **权限申请**: 用户申请某文件的下载/预览权限，管理员审批。
  - **下载审批**: 敏感文件下载需二次审批。

### 2.2 文档转换与增强预览 (Document Conversion)
- **现状**: 仅支持 PDF 和 纯文本 的原生预览。Office 文档需下载查看。
- **建议**: 引入 `LibreOffice` 或 `Pandoc` 容器化服务。
  - 上传 Office 文档 (`.docx`, `.pptx`, `.xlsx`) 时，后台异步转换为 PDF。
  - 前端统一使用 PDF 预览器查看所有文档。

### 2.3 安全增强 (Security Enhancements)
- **水印 (Watermark)**:
  - 在预览 PDF/图片时，动态叠加“用户名 + 时间 + IP”水印，防止截屏泄露。
- **分享链接 (Share Links)**:
  - 生成带有效期的外部访问链接 (如 `uuid` 链接)，允许非项目成员临时访问特定文件。
- **下载限制**:
  - 限制单人每日下载量，或对特定敏感级别文件禁止下载（仅预览）。

### 2.4 任务与协作 (Tasks & Collaboration)
- **现状**: 有基础的 `ProjectTask` 模型。
- **建议**:
  - **看板视图**: 前端增加拖拽式看板 (Kanban) 管理任务状态。
  - **甘特图**: 展示任务与里程碑的时间线。
  - **文件关联**: 任务可关联具体的文件或目录。

### 2.5 性能优化 (Performance)
- **大文件分片上传**: 目前是单次 POST 上传，大文件可能超时。建议改为分片上传 (Chunked Upload)。
- **异步索引**: 目前上传时同步提取文本 (`_extract_text`)，大 PDF 可能阻塞请求。建议引入 `Celery` 或 `Django Q` 进行异步处理。

## 3. 部署建议
- 由于 GitHub 连接问题，建议在内网环境搭建 `GitLab` 或 `Gitea` 进行代码托管。
- 生产环境建议使用 `Nginx` 反向代理静态文件与后端 API。
- 数据库建议迁移至 `PostgreSQL` 以获得更好的并发性能与全文检索支持 (可利用 PG 的 `tsvector`)。
